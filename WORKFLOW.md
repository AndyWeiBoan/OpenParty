# OpenParty — Research Workflow

> 這份文件定義 OpenParty 每個 Milestone 的自我研究工作方法。
> 每次展開一個新的研究課題，都照這個流程走。

---

## 核心原則

- **每個課題都要有最小化驗收標準**，不能模糊地說「研究一下」
- **先設計再實作**，不要邊做邊想
- **失敗和問題都要記錄**，是下一個課題的養分
- **每個 Milestone 一個資料夾**，研究成果完整封存

## 研究流程（每個課題重複執行）

### Step 1 — 選題

從 `ROADMAP.md` 找出目前 Milestone 中**最重要、最不確定**的課題。

選題原則：
- 優先選「如果搞錯方向會浪費最多時間」的課題
- 一次只選一個課題，做完再選下一個
- 如果一個課題太大，拆成更小的子課題

---

### Step 2 — 調研（如果需要）

如果課題本身有足夠把握，跳過這步直接到 Step 3。

需要調研時：
- 上網搜尋現有方案、競品做法、社群討論
- 整理成幾個可選的方向（不要只找一個答案）
- 每個方向列出優點、缺點、適用場景

輸出：在 `poc-design.md` 的「調研結果」區塊記錄

---

### Step 3 — 設計 PoC → `poc-design.md`

明確定義這次要驗證什麼，以及怎麼知道成功了。

`poc-design.md` 必須包含：

```markdown
## 課題
一句話說清楚要驗證什麼假設

## 背景
為什麼這個課題重要？不驗證會有什麼風險？

## 調研結果（如果有）
方向 A：...優點 / 缺點
方向 B：...優點 / 缺點

## 選定方向
選哪個，為什麼

## PoC 設計
最小化的實驗設計，只驗證核心假設
不需要完整功能，只需要能回答「這個方向可不可行」

## 驗收標準
- [ ] 具體可量測的標準 1
- [ ] 具體可量測的標準 2
- [ ] ...
必須是「跑一次就知道過沒過」的標準，不能模糊

## 不在範圍內
明確列出這次不驗證的東西，避免 scope creep
```

---

### Step 4 — 實作 → `poc-implement.md` + `src/`

按照 `poc-design.md` 的設計實作，記錄思路。

`poc-implement.md` 必須包含：

```markdown
## 實作思路
用幾句話說清楚做了什麼，讓未來的自己看得懂

## 檔案結構
src/
├── 檔案 A  ← 做什麼的
└── 檔案 B  ← 做什麼的

## 關鍵設計決策
做了哪些非顯而易見的選擇，為什麼這樣選

## 如何執行
step by step 的執行指令，確保任何人都能重現
```

實作原則：
- **最小化**：只寫驗證假設所需的程式碼
- **可重現**：別人 clone 下來跑得起來
- **不要過度設計**：這是研究用的，不是產品

---

### Step 5 — 選定最優方案

如果 Step 3 設計了多個方向，在這裡選出最終要採用的方案。

選定標準：
- 驗收標準通過了幾項？
- 實作複雜度如何？
- 對後續 Milestone 的影響？

在 `poc-result.md` 記錄選定理由。

---

### Step 6 — 紀錄結果 → `poc-result.md`

```markdown
## 結論
一句話：這個課題的答案是什麼

## 驗收標準結果
- [x] 標準 1 — 通過，原因
- [x] 標準 2 — 通過，原因
- [ ] 標準 3 — 未通過，原因

## 選定方案
最終選了哪個方向，為什麼

## 對 ROADMAP 的影響
這個結果有沒有改變後續 Milestone 的計畫？
需不需要更新 ROADMAP.md？

## 下一步
這個課題結束後，建議研究哪個課題
```

---

### Step 7 — 紀錄問題 → `poc-log.md`

研究過程中遇到的所有問題，不管大小都記錄。

```markdown
## [日期] 問題描述
**症狀**：看到了什麼
**原因**：為什麼發生
**解法**：怎麼解決的
**教訓**：下次要注意什麼
```

這個文件是最有價值的資產之一，不要省略。

## 更新規則

- `ROADMAP.md`：每完成一個課題後，**必須**回去更新，標記完成狀態
- `WORKFLOW.md`：流程本身有問題時更新，不要頻繁改動
- 每個 `poc-*.md`：完成當下寫，不要事後補寫（記憶會失真）

---

## 發現缺失階段時的處理方式

研究過程中，可能發現原本的 Milestone 順序有缺漏。

**例子**：原本計畫是 `A → C → D`，研究 C 的過程中發現其實需要先做 B，否則 C 的方向會走錯。

**處理流程**：

1. 先在當前的 `poc-log.md` 記錄這個發現
   ```markdown
   ## [日期] 發現缺失階段
   **發現**：研究 C 的過程中發現 B 是前置條件
   **影響**：不先做 B，C 的方向可能走錯
   **決定**：插入 B 階段
   ```

2. 回去更新 `ROADMAP.md`，在正確的位置插入新階段
   ```
   原本：A → C → D
   更新後：A → B（新增）→ C → D
   ```

3. 為新階段建立對應的資料夾，從 Step 1 重新開始走流程

4. 新階段研究完成後，再繼續原本的 C

**原則**：寧可暫停當前課題、補做前置研究，也不要帶著不確定的假設繼續往前走。

## 每個階段結束的 ROADMAP 回寫

每個 Milestone 的所有課題研究完畢後，必須更新 `ROADMAP.md`：

1. 將該 Milestone 的狀態改為 `完成 ✅`，並標記完成日期
2. 將 `poc-result.md` 的結論摘要寫回 ROADMAP 對應的區塊
3. 如果研究結果改變了後續 Milestone 的計畫，同步修改後續的描述
4. 更新 Milestone 總覽的進度條

```
更新前：
M1  產品方向決策  ░░░░░░░░░░░░  進行中 🔄

更新後：
M1  產品方向決策  ████████████  完成 ✅  2026/xx/xx
```

**ROADMAP.md 是唯一的事實來源**，任何人看它都應該能知道專案目前在哪裡。
